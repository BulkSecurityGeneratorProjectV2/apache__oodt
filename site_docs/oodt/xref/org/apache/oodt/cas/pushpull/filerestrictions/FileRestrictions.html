28  /**
29   * This class allows the creation of restrictions for files and directories created below an actual directory which is passed
30   * into the constructor.  These restriction are loaded by passing a FileInputStream which contains a XML File
31   * into the  method and can be tested against by using the  method.
32   * 
33   * <pre>
34   * The XML file schema is:
35   *	{@literal <root>
36   *	   <variables>
37   *	      <variable name="variable-name">
38   *	         <type>INT-or-STRING</type>
39   *	         <value>variable-value</value>
40   *	         <precision>
41   *	            <locations>number-of-fill-locations</locations>
42   *	            <fill>fill-value</fill>
43   *	            <side>front-or-back</side>
44   *	         </precision>
45   *	      </variable>
46   *	      ...
47   *	      ...
48   *	   </variables>
49   *
50   *	   <methods>
51   *	      <method name="method-name">
52   *	         <args>
53   *	            <arg name="argument-name">
54   *	               <type>INT-or-STRING</type>
55   *	            </arg>
56   *	            ...
57   *	            ...
58   *	         </args>
59   *	         <action>method-behavior</action>
60   *	      </method>
61   *	      ...
62   *	      ...
63   *	   </methods>
64   *   
65   *	   <dirstruct name="root-directory-name">
66   *	      <nodirs/>
67   *	      <nofiles/>
68   *	      <file name="file-name"/>
69   *	      <dir name="directory-name">
70   *	         <nodirs/>
71   *	         <nofiles/>
72   *	         <file name="file-name"/>
73   *	         <dir name="directory-name">
74   *	            ...
75   *	            ...
76   *	         </dir>
77   *	      </dir>
78   *	      ...
79   *	      ...
80   *	   </dirstruct>
81   *	</root>}
82   * </pre>
83   *
84   * <i>{@literal <variables>}</i> and <i>{@literal <methods>}</i> can be created in this XML file so that they can be used in the <i>{@literal <dirstruct>}</i>
85   * portion of the XML file.  These <i>{@literal <variables>}</i> and <i>{@literal <methods>}</i> can be used inside the <i>{@literal <dir>}</i> and
86   * <i>{@literal <file>}</i> elements within the <i>{@literal <dirstruct>}</i> element to allow for varrying directory and file names beyond the capability
87   * of regular expressions (which are also allowed).  
88   * 
89   * <h2>VARIABLES (OPTIONAL):</h2>
90   * Let's start with describing the <i>{@literal <variables>}</i> portion of the XML file.  As many <i>{@literal <variable>}</i> elements as you would like
91   * can be specified inside the <i>{@literal <variables>}</i> tag.  The <i>{@literal <variable>}</i> element must have a parameter, 'name', which is the name
92   * of this <i>{@literal <variable>}</i>.  Every <i>{@literal <variable>}</i> is a global variable (that is, global in the scope of the XML file it is declared
93   * in -- it is not usable in other XML file, unless redeclared) so variable names are unique (however, are case sensitive) so thus a name can only
94   * be applied to one <i>{@literal <variable>}</i>.  Within the <i>{@literal <variable>}</i> element there are three possible sub-elements that can be included.
95   * <i>{@literal <type>}</i> and <i>{@literal <value>}</i> are required and <i>{@literal <precision>}</i> is optional.  <i>{@literal <type>}</i> can be either (and it must be
96   * in all UPPERCASE) INT or STRING (sorry, floating point numbers are not supported as of yet).  This specifies what type of value will be given in 
97   * <i>{@literal <value>}</i>.  This allows you to both use numerical values as either an integer or a string.  <i>{@literal <precision>}</i> can also be specified
98   * for each <i>{@literal <variable>}</i>.  This allows you to insure that an integer or string will take up a certain amount of space.  This is especially
99   * useful when dealing with dates.  For instance, say you had the following in your XML file: 
100  * 
101  * <pre>
102  *	{@literal <variable name="myVariable">
103  *	   <type>INT</type>
104  *	   <value>3</value>
105  *	</variable>}
106  * </pre>
107  *
108  * When myVariable was finally returned it would look like 3, however many times for dates you would like 03 returned.  You can specify this by adding
109  * precision to the following XML:
110  *
111  * <pre>
112  *	{@literal <variable name="myVariable">
113  *	   <type>INT</type>
114  *	   <value>3</value>
115  *	   <precision>
116  *	      <locations>2</locations>
117  *	      <fill>0</fill>
118  *	      <side>front</side>
119  *	   </precision>
120  *	</variable>}
121  * </pre>
122  *
123  * This insures that the number is always printed with 2 digits and if the number does not take up 2 digits worth of space a fill value 0 will be added
124  * to the front side of the integer, thus, in this example would give us 03.  Note: <i>{@literal <value>03</value>}</i> would NOT accomplish the same!!!!
125  * 
126  * <h2>METHODS (OPTIONAL):</h2>
127  * Next let's look at the <i>{@literal <methods>}</i> portion of the XML file.  <i>{@literal <method>}</i> elements must have a 'name' parameter, which is the name of the
128  * <i>{@literal <method>}</i>.  Every <i>{@literal <method>}</i> is also global in the same way as is every <i>{@literal <variable>}</i> and are also case-sensitive, thus method 
129  * names must be unique.  A <i>{@literal <method>}</i> element may contain an <i>{@literal <args>}</i> sub-element, however this is optional and only needed if the method
130  * is to take any arguments.  If an <i>{@literal <args>}</i> element is given, then it should contain at least one <i>{@literal <arg>}</i> element.  A <i>{@literal <method>}</i>
131  * may contain as many <i>{@literal <arg>}</i> elements as it needs.  What is being specified by a <i>{@literal <method>}</i> element is what would be known in java code 
132  * as the method signature.  Thus all we are going to specify is each argument's name and type.  Thus each <i>{@literal <arg>}</i> element must contain
133  * a 'name' parameter, which is the name of the argument and must contain a <i>{@literal <type>}</i> sub-element, so it is known how to treat the arguments when 
134  * the method is used within the <i>{@literal <dirstruct>}</i> section of the XML file.  Another sub-element, which is required, for the <i>{@literal <method>}</i>
135  * element is the <i>{@literal <action>}</i> element.  This element contains the behavior of the <i>{@literal <method>}</i>.  Before going into detail about what can be
136  * placed within the <i>{@literal <action>}</i> element let's first cover some syntax requirements for the XML file.
137  * 
138  * <h2>SYNTAX REQUIREMENTS:</h2>
139  * When a <i>{@literal <variable>}</i> is used it must be preceded by $ and inclosed in {} (e.g. ${myVariable}).<br>
140  * When a <i>{@literal <method>}</i> is used it must be preceded by % and end with () (e.g. %myMethod(), however if arguments are given then %myMethod(12,9)).<br>
141  * When a <i>{@literal <method>}</i> argument (<i>{@literal <arg>}</i> element) is used is must be preceded by $ (e.g. $myArg).<br>
142  * When a literal integer is used it must be preceded by # (e.g. #234).<br>
143  * When a literal string is used it must be inclosed in " (e.g. "my age is 56 -- no not really").<br>
144  *	
145  *	<p>NOTE: When passing arguments into methods the string and integer literal rules do not need to be followed because you have already defined what each
146  *	   argument type should be and they will be evaluated as such.</p>
147  *	<p>NOTE*: Also note that at present a <i>{@literal <variable>}</i> cannot be passed as an argument to the methods.  Just use the <i>{@literal <variable>}</i> where needed
148  *	   inside the <i>{@literal <action>}</i> element.  This feature should hopefully be added in a later release.</p>
149  *
150  * <h2>METHOD'S ACTION ELEMENT USAGE:</h2>
151  * The <i>{@literal <action>}</i> element will evaluate expressions that contain both integers and strings.  It obeys the rules of mathematical precedence and will
152  * also handle parentheses.  It also, like Java, still follows the order of precedence when strings are present.  That is, if you have the expression:<br>
153  * &nbsp&nbsp	<i>#2+#4+" years old, going on "+#2+#4</i><br>
154  * It would evaluate to:<br>
155  * &nbsp&nbsp	<i>6 years old, going on 24</i><br>
156  * You may use any <i>{@literal <variable>}</i> declared within the same XML file and may also use any argument (<i>{@literal <arg>}</i> element) declared within that 
157  * <i>{@literal <method>}</i>.  Also string and integer literals may be used.  Currently the only operators supported are +,-,*,/ (which are respectively:
158  * addition, subtraction, multiplication, and division).  Parentheses, (), and embedded parentheses, (()()), are also all allowed.
159  * 
160  * <h2>DIRSTRUCT:</h2>
161  * The final section of the XML file is the actual main purpose of the XML file.  This is the XML that controls which directories the crawler will be allowed
162  * to crawl and which files will be allowed.  The <i>{@literal <dirstruct>}</i> element requires a 'name' parameter which is the path to the root
163  * directory that is to be considered (that is, all other directories below the given directory are unimportant and will not be crawled).  You want your root
164  * directory path to stop at the first directory in which you are interested in more than one of its sub-directories or want file(s) inside it.
165  * For example, let say we want to crawl a remote site that has the following directory structure:
166  * 
167  * <pre>
168  *	-parent
169  *	   -child1
170  *	      -grandChild1
171  *	         -greatGrandChild1
172  *              -file1
173  *	         -greatGrandChild2
174  *	      -grandChild2
175  *	      -file1
176  *	   -child2 
177  *	      -file1
178  * 	   -child3
179  * 	      -file1
180  *	      -file2
181  * 	      -grandChild1
182  * 	         -file1
183  *	         -file2
184  * 	   -child4
185  * </pre>
186  * 
187  * Now, say, we only are interested in directories and files below the two shown 'grandChild1' directories.  This would mean that for our <i>{@literal <dirstruct>}</i>
188  * 'name' parameter we would put name="parent".  This is because we need access to both 'child1' and 'child3' subdirectories.  Now in order to avoid crawling
189  * 'child2' and 'child4' directories we have to specify <i>{@literal <dir>}</i> elements.  This would give us the following XML:
190  * 
191  * <pre>
192  * 	{@literal <dirstruct name="/parent">
193  *	   <dir name="child1"/>
194  *	   <dir name="child3"/>
195  *	</dirstruct>}
196  * </pre>
197  *
198  * This would restrict the directories allowed under 'parent' to only be directories with names either 'child1' or 'child3', all other directory names will
199  * be rejected.  However, more must be added to this example because we have not yet specified any restrictions on files allowed beneath 'parent',
200  * we have to add the <i>{@literal <nofiles/>}</i> element:
201  * 
202  * <pre>
203  *	{@literal <dirstruct name="/parent">}
204  *	   <i><b>{@literal <nofiles/>}</b></i>
205  *	   {@literal <dir name="child1"/>
206  *	   <dir name="child3"/>
207  *	</dirstruct>} 
208  * </pre>
209  *
210  * Now the only thing acceptable below parent is 'child1' and 'child3'.  We have to still further our restrictions under 'child1' and 'child3'.  Since under 
211  * 'child1' we only want 'grandChild1' we would have to make another <i>{@literal <dir>}</i> element and also add a <i>{@literal <nofiles/>}</i> element:
212  * 
213  * <pre>
214  *	{@literal <dirstruct name="/parent">
215  *	   <nofiles/>
216  *	   <dir name="child1">}
217  *	      <i><b>{@literal <nofiles/>
218  *	      <dir name="grandChild1"/>}</b></i>
219  *	   {@literal </dir>
220  *	   <dir name="child3"/>
221  *	</dirstruct>}  
222  * </pre>
223  * 
224  * We have to do the same also for 'child3', giving us:
225  * 
226  * <pre>
227  *	{@literal <dirstruct name="/parent">
228  *	   <nofiles/>
229  *	   <dir name="child1">
230  *	      <nofiles/>
231  *	      <dir name="grandChild1"/>
232  *	   </dir>
233  *	   <dir name="child3">}
234  *	      <i><b>{@literal <nofiles/>
235  *	      <dir name="grandChild1"/>}</b></i>
236  *	   {@literal </dir>
237  *	</dirstruct>}  
238  * </pre>
239  *
240  * From the example directory structure above, with this XML file specified, that directory structure would be limited to:
241  * 
242  * <pre>
243  *	-parent
244  *	   -child1
245  *	      -grandChild1
246  *	         -greatGrandChild1
247  *              -file1
248  *	         -greatGrandChild2
249  * 	   -child3
250  * 	      -grandChild1
251  * 	         -file1
252  *	         -file2
253  * </pre>
254  * 
255  * Say we now decide that we only want files below the two 'grandChild1' directories -- that is, no directories.  So we would change or XML by adding 
256  * in the <i>{@literal <nodir/>}</i> element:
257  * 
258  * <pre>
259  *	{@literal <dirstruct name="/parent">
260  *	   <nofiles/>
261  *	   <dir name="child1">
262  *	      <nofiles/>
263  *	      <dir name="grandChild1">}
264  *	          <i><b>{@literal <nodirs/>}</b></i>
265  *	      {@literal </dir>
266  *	   </dir>
267  *	   <dir name="child3">
268  *	      <nofiles/>
269  *	      <dir name="grandChild1">}
270  *	         <i><b>{@literal <nodirs/>}</b></i>
271  *	      {@literal </dir>
272  *	   </dir>
273  *	</dirstruct>}   
274  * </pre>
275  *
276  * Which now restricts our directory structure to:
277  * 
278  * <pre>
279  *	-parent
280  *	   -child1
281  *	      -grandChild1
282  * 	   -child3
283  * 	      -grandChild1
284  * 	         -file1
285  *	         -file2
286  * </pre>
287  * 
288  * Let's further specify now that we only want 'file1' in the '/parent/child3/grandChild1' directory.  This would change the XML to:
289  * 
290  * <pre>
291  *	{@literal <dirstruct name="/parent">
292  *	   <nofiles/>
293  *	   <dir name="child1">
294  *	      <nofiles/>
295  *	      <dir name="grandChild1">
296  *	          <nodirs/>
297  *	      </dir>
298  *	   </dir>
299  *	   <dir name="child3">
300  *	      <nofiles/>
301  *	      <dir name="grandChild1">
302  *	         <nodirs/>}
303  *	         <i><b>{@literal <file name="file1"/>}</b></i>
304  *	      {@literal </dir>
305  *	   </dir>
306  *	</dirstruct>}
307  * </pre>
308  *
309  * Our new allowed directory structure would now be:
310  * 
311  * <pre>
312  *	-parent
313  *	   -child1
314  *	      -grandChild1
315  * 	   -child3
316  * 	      -grandChild1
317  * 	         -file1
318  * </pre>
319  * 
320  *	NOTES:
321  * 	-You would not want to use the <i>{@literal <nofiles/>}</i> and <i>{@literal <file>}</i> elements in the same directory (same goes for the
322  * 	   <i>{@literal <nodirs/>}</i> and <i>{@literal <dir>}</i> elements) because you would be specifying that you don't want any files in that
323  * 	   directory, and then contradict yourself by specifying a <i>{@literal <file>}</i> element that is okay to have.  The <i>{@literal <file>}</i> 
324  *	   element states that no other file but the file I specified is allowed.  The only exception is if you have two or more <i>{@literal <file>}</i> 
325  *	   elements in the same directory -- this is allowed.  It follows the same rules as the <i>{@literal <dir>}</i> element in the example 
326  *	   given above where only 'child1' and 'child3' were allow.  The two don't cancel each other out. 
327  * 
328  * <h2>ADVANCED USAGES OF DIRSTRUCT:</h2>
329  * Regular expressions are allowed in the 'name' parameter of both <i>{@literal <dir>}</i> and <i>{@literal <file>}</i> elements.  Also any 
330  * <i>{@literal <method>}</i> or <i>{@literal <variable>}</i> element declared can be used within the 'name' parameter of both <i>{@literal <dir>}</i> 
331  * and <i>{@literal <file>}</i> elements.  There are also several predefined variables that can be used.
332  * 
333  * <h4>REGULAR EXPRESSIONS:</h4>
334  * The regular expressions are parsed by the <a href=http://java.sun.com/j2se/1.5.0/docs/api/java/util/regex/Pattern.html>Pattern</a> class 
335  * (See its documentation on rule for specifying regular expressions).  Here is an example use of a regular expression:
336  * 
337  * <pre>
338  *	{@literal <dirstruct name="/.../temp/test">
339  *	   <nofiles/>
340  *	   <dir name="\d{4}-\d{2}-\d{2}">
341  *	      <nodirs/>
342  *	   </dir>
343  *	</dirstruct>}
344  * </pre>
345  *
346  * This would restrict the directory files in directories below /.../temp/test to only directories whose names are dates of the
347  * format: YYYY-MM-DD.
348  * 
349  * <h4>PREDEFINED DATE VARIABLES:</h4>
350  * There are several predefined date variables than can be put as the <i>{@literal <value>}</i> of a <i>{@literal <variable>}</i> and then used.
351  * 
352  * These variables are:
353  * <pre>
354  * 	[DATE.DAY]	- day of today's date
355  * 	[DATE.MONTH]	- month of today's date
356  * 	[DATE.YEAR]	- year of today's date
357  * 	[DATE-N.DAY]	- the day of the date N days ago
358  * 	[DATE-N.MONTH]	- the month of the date N days ago
359  * 	[DATE-N.YEAR]	- the year of the date N days ago
360  * 	[DATE+N.DAY]	- the day of the date N days from now
361  * 	[DATE+N.MONTH]	- the month of the date N days from now
362  * 	[DATE+N.YEAR]	- the year of the date N days from now
363  * 
364  * 	-sorry, no DayOfYear implemented yet -- hopefully in a later release
365  * </pre>
366  * 
367  * Usage:
368  * <pre>
369  * 	{@literal <root>
370  * 	   <variables>
371  * 	      <variable name="todaysDay">
372  * 	         <type>INT</type>
373  * 	         <value>[DATE.DAY]</value>
374  *	         <precision>
375  *	            <locations>2</locations>
376  *	            <fill>0</fill>
377  *	            <side>front</side>
378  *	         </precision>
379  *	      </variable>
380  *	   </variabls>
381  *	   <dirstruct name="/path/to/parent/dir">
382  *	      <nofiles/>
383  *	      <dir name="MyFiles">
384  *	         <nodirs/>
385  *	         <file name="MyPaper_${todaysDay}"/>
386  *	      </dir>
387  *	   </dirstruct>
388  *	</root>}
389  *  </pre>
390  *  
391  *  This would allow only a file in /path/to/parent/dir/MyFiles which had the name which started with MyPaper_ and ended with the
392  *  day of the current day of the month.  For example, if to days date was 03/23/2005, then the file name allowed would be
393  *  MyPaper_23.
394  * 
395  * <h4>METHOD AND VARIABLE USAGE IN DIRSTRUCT:</h4>
396  * Here is an example of using <i>{@literal <variables>}</i> and <i>{@literal <methods>}</i>:
397  * 
398  * <pre>
399  *	{@literal <root>
400  *	   <variables>
401  *	      <variable name="DAY">
402  *	         <type>INT</type>
403  *	         <value>[DATE.DAY]</value>
404  *	         <precision>
405  *	            <locations>2</locations>
406  *	            <fill>0</fill>
407  *	            <side>front</side>
408  *	         </precision>
409  *	      </variable>
410  *	      <variable name="MONTH">
411  *	         <type>INT</type>
412  *	         <value>[DATE.MONTH]</value>
413  *	         <precision>
414  *	            <locations>2</locations>
415  *	            <fill>0</fill>
416  *	            <side>front</side>
417  *	         </precision>
418  *	      </variable>
419  *	      <variable name="YEAR">
420  *	         <type>INT</type>
421  *	         <value>[DATE.YEAR]</value>
422  *	      </variable>
423  *	   </variables>
424  *   
425  *	   <methods>
426  *	      <method name="ADD">
427  *	         <args>
428  *	            <arg name="1">
429  *	               <type>INT</type>
430  *	            </arg>
431  *	         </args>
432  *	         <action>"THE_YEAR_PLUS_"+$1+": "+(${YEAR}+$1)</action>
433  *	      </method>
434  *	      <method name="HOW_OLD_AM_I">
435  *	         <action>${YEAR}-#1984</action>
436  *	      </method>
437  *	      <method name="DATE">
438  *	         <action>${YEAR}+"-"+${MONTH}+"-"+${DAY}</action>
439  *	      </method>
440  *	   </methods>
441  *
442  *	   <dirstruct name="/path/to/parent/dir">
443  *	      <nofiles/>
444  *	      <dir name="AGE_%HOW_OLD_AM_I()"/>
445  *	      <dir name="DATE">
446  *	         <nodirs/>
447  *	         <file name="%ADD(5)"/>
448  *	      </dir>
449  *	   </dirstruct>
450  *	</root>}
451  * </pre>
452  *
453  * This would accept only the directories under /path/to/parent/dir which had the name (given today is 9/7/2007) 'AGE_23' or '2007-09-07'.
454  * This would allow any file or directory in under 'AGE_23', but would only allow a file with the name 'THE_YEAR_PLUS_5: 2012' in the 
455  * directory '2007-09-07'.
456  *  
457  * 
458  * @author bfoster
459  *
460  */
461 public class FileRestrictions {
462 
463     private FileRestrictions() throws InstantiationException{
464         throw new InstantiationException("Don't construct FileRestrictions!");
465     }
466 
467     /**
468      * 
469      * @param path
470      * @return The initial cd directory which needs to be changed to (in order
471      *         to take care of possible auto-mounted directories)
472      */
473     public static boolean isAllowed(ProtocolPath path, VirtualFile root) {
474         return (isValidPath(path) && isAllowed(new VirtualFile(path
475                 .getPathString(), path.isDirectory()), root));
476     }
477 
478     public static boolean isAllowed(VirtualFile file, VirtualFile root) {
479         boolean vfDoesNotExist = false, lastFileIsDir = file.isDir();
480         VirtualFile vf;
481         while ((vf = root.getChildRecursive(file)) == null) {
482             vfDoesNotExist = true;
483             lastFileIsDir = file.isDir();
484             file = file.getParentFile();
485             if (file == null) {
486                 break;
487             }
488         }
489         return !(file == null || (vfDoesNotExist && ((lastFileIsDir && !vf
490                 .allowNewDirs()) || (!lastFileIsDir && !vf.allowNewFiles()))));
491     }
492 
493     private static boolean isValidPath(ProtocolPath path) {
494         return path != null && !path.getFileName().equals(".")
